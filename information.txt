Function-call flow: GooseService calls start(iface) which creates/configures/activates a pcap handle filtered to ether proto 0x88B8 and launches loop(); registerStream/clearStreams maintain the routing map (empty gocbRef registers a catch-all); the capture loop() calls pcap_next_ex per frame, runs the anonymous-namespace decoder goose_decode (which skips the Ethernet/VLAN header, validates EtherType 0x88B8 and the GSE header, parses the IECGoosePdu via the BER helpers ber_read_len/ber_find to extract gocbRef [0x80], the 8-octet UtcTime t [0x84], and the first BOOLEAN [0x83] inside allData [0xAB]), then looks up the streamId for that gocbRef and, on a match, builds an SpscMessage (type SPSC_VALUE_BOOLEAN) and calls SpscBridge::instance().pushOutbound(msg); stop() breaks the pcap loop and joins the thread. Sequence-wise it is GooseService::sv_goose_rx_start -> GooseReceiver::start -> capture loop -> pcap_next_ex -> goose_decode (BER parse) -> gocbRef route lookup -> SpscBridge::pushOutbound -> Teammate App (via sv_spsc_pop_outbound), with permissive decoding and per-stage stat counters.


File(s): GooseService.cc, GooseService.h. This module is the singleton holder and C ABI facade for the GOOSE TX schedulers and the single RX, with no public class — instead an anonymous-namespace struct Service (a std::mutex, a std::unordered_map<uint16_t, std::unique_ptr<GooseTxScheduler>> tx, a parallel std::unordered_map<uint16_t, GooseEncoderConfig> txConfig, and a single std::unique_ptr<GooseReceiver> rx) accessed through a function-local svc() singleton, plus a copy_ref string-copy helper. Function-call flow: the extern "C" API (declared in GooseService.h, dispatched from PubWsServer's goose_* commands) drives everything under the mutex — sv_goose_configure_tx fills a GooseEncoderConfig and stores it in txConfig; sv_goose_start_tx looks up that config, constructs a GooseTxScheduler, calls setConfig/setSettings and start() (idempotent if already running), and moves it into the tx map; sv_goose_stop_tx/sv_goose_stop_all_tx move schedulers out and destroy them outside the lock (so stop()/join() can't deadlock); sv_goose_rx_start lazily constructs the single GooseReceiver and calls start(iface), sv_goose_rx_stop moves it out to join outside the lock, and sv_goose_rx_register/sv_goose_rx_clear forward to the receiver's routing map; sv_goose_get_stats reads framesSent/framesFailed from the per-stream scheduler and framesSeen/framesPushed from the receiver. Sequence-wise it is PubWsServer goose_* command -> sv_goose_* (mutex-locked) -> GooseTxScheduler (TX, per stream) / GooseReceiver (RX, single) lifecycle + stats, acting as the central registry that wires the GOOSE encoder, scheduler, and receiver to the WebSocket layer and SpscBridge.

This module is an embeddable C++ library (namespace substation::) that lets a teammate's substation simulator (e.g. Modelica) run the publisher's core capabilities in-process instead of over WebSocket; it does not duplicate backend code but forwards to native/src. ACTOR BOUNDARY (read this FIRST when drawing the DFD — there are exactly TWO actors, and the kit is only a thin bridge between them, NOT a process of its own): ACTOR A = the EXISTING publisher backend under native/src (SvPublisherInstance, EquationProcessor, SvEncoder, SharedBuffer, the writer thread, PcapTx, and the GOOSE stack) — it OWNS ALL sample-generation, encoding, scheduling, and transmission; the kit adds NO new transmit path. ACTOR B = Shivani's substation simulator application — its own main() links this kit and is the ONLY source of LIVE runtime data; it is NOT a separate node to be drawn as a remote app, because everything (ACTOR A backend + ACTOR B main + the kit) compiles into ACTOR B's SINGLE process. THE KIT'S PURPOSE IS LIVE DATA ONLY — it is the simulator's front-end to the shared backend. EQUATION-DRIVEN SV IS NOT A KIT FLOW: in the product, waveform equations are configured by the EXISTING JS FRONTEND publisher (browser UI -> WebSocket -> PubWsServer -> the same PublisherController backend). That JS frontend is a SEPARATE, interchangeable front-end to the SAME backend; in the DFD it is its own source box (JS frontend -> PubWsServer -> PublisherController), and the equation waveform arrow originates THERE, never from the kit box. The kit's public API does re-export setEquations/add/configure/start (because it forwards to the same shared PublisherController), but those are convenience forwarders, not the kit's role — so the KIT BOX in the DFD should show ONLY the live-data nodes (net::open, add/configure, setExternalMode, init()->worker+MPMC, submitMagnitude/submitBoolean, GOOSE config/lifecycle) and must NOT contain an equation/formula producer node. EQUATIONS ARE A SETUP-TIME CONFIG INPUT, NOT A RUNTIME/SENDING NODE (this is what confuses auto-DFD tools): substation::publisher::setEquations(id, "Ia:100*sin(2*PI*50*t)|...") is a ONE-TIME configuration call passing a formula STRING; it is consumed once by ACTOR A's EquationProcessor inside prebuildFrames(). The simulator does NOT 'send equations' and there is NO equation producer/streaming node — in the DFD, equation mode is merely a static config arrow into prebuildFrames; thereafter its only data flow is prebuilt-frame-bytes -> SharedBuffer -> writer -> NIC, entirely inside ACTOR A. Per publisher you pick EITHER equation mode OR live mode via setExternalMode (mutually exclusive). So the kit drives exactly TWO live flows (both from ACTOR B, the simulator): (1) LIVE/EXTERNAL SV — ACTOR B feeds magnitudes that ACTOR A re-encodes into the SAME SharedBuffer frames at TX time; uses the kit's in-process MPMC ingest worker + the SvPublisherInstance live-value store; does NOT use the SPSC bridge. (2) GOOSE publish/subscribe — boolean breaker/trip states through ACTOR A's GooseTxScheduler/GooseReceiver, which use the SpscBridge as their (and the only) value channel. A THIRD flow, EQUATION-DRIVEN SV, exists in the shared backend but is NOT a kit flow — it is owned by the SEPARATE JS frontend (JS frontend -> PubWsServer -> PublisherController -> SvPublisherInstance::prebuildFrames using the configured formula -> SharedBuffer -> writer -> NIC), purely inside ACTOR A and driven by the browser UI, with no simulator/kit runtime input. Draw equation-SV only if you are also drawing the JS-frontend deployment; in a simulator-only DFD it does not appear. LIVE-DATA NODE CHAIN (the exact nodes+arrows the DFD must show, the part auto-tools keep dropping): [ACTOR B: simulator main()] --calls--> [publisher::init() NODE] --spawns--> [ingest worker thread + MPMC queue NODE] (both created inside ACTOR B's process). Separately, [ACTOR B: simulator worker threads] --submitMagnitude/submitBoolean (multi-producer try_push)--> [MPMC queue]. Then [MPMC queue] --pop--> [single ingest worker (routeMessage)] --and from here every outgoing arrow lands on EXISTING ACTOR A nodes--> for SV magnitude: --> [SvPublisherInstance::setLiveValue] --> [m_liveValue store] --> [writer thread reencodeFrame] --> [SharedBuffer frame bytes (rewritten in place)] --> [PcapTx npcap_send_packet] --> [NIC]; for GOOSE boolean: --> [goose::setValue] --> [SpscBridge] --> [GooseTxScheduler] --> [NIC]. KEY ARROWS: the arrow INTO init() comes FROM the simulator main(); init()'s outgoing arrow goes ONLY to the worker+MPMC; the worker's outgoing arrows go ONLY to the existing ACTOR A nodes (never to a brand-new transmit path). LIVE-INGEST mechanism in code: substation::publisher::init(queueCapacity) lazily allocates a rigtorp::MPMCQueue<IngestMsg> (IngestMsg = anonymous-namespace POD: id, channel, isBoolean, quality, magnitude, boolean, timestampNs) and spawns ONE consumer thread (ingestWorkerLoop); shutdown() flips the atomic g_ingestRunning, joins the worker, drains, and frees the queue (reset() calls shutdown() first so the worker stops before any publisher is destroyed). submitMagnitude(id, channel, value, quality, t_ns) and submitBoolean(streamId, value, quality, t_ns) are both non-blocking try_push (multi-producer). The single worker pops each message (routeMessage) and routes by type: magnitude -> PublisherController::getPublisher(id)->setLiveValue(channel, magnitude, quality) (SharedBuffer path, no SPSC); boolean -> substation::goose::setValue(streamId, ...) (one SPSC_VALUE_BOOLEAN into SpscBridge — GOOSE-only). substation::publisher::setExternalMode(id, true) flips that publisher to SourceMode::External so the writer thread's reencodeFrame() reads the live store. Public surface: substation::publisher::add/configure/setEquations/start/stop/reset/isRunning/count/removeAll/lastError PLUS init/shutdown/setExternalMode/submitMagnitude/submitBoolean (forwarding to PublisherController + SvPublisherInstance; publisher::Config mirrors PublisherConfig byte-for-byte, static_assert-checked); substation::goose::configureTx/setValue/startTx/stopTx/stopAllTx/rxStart/rxStop/rxRegister/rxClear/getStats/popDecoded (goose::setValue pushes a single SPSC_VALUE_BOOLEAN into SpscBridge to feed the scheduler — GOOSE-only); substation::net::* -> npcap_* (PcapTx); and stateless encoder helpers substation::sv::* -> sv_encoder_*, substation::goose::encodeFrame -> goose_encode_frame, substation::ber::* -> ber_encode_* (all static_assert-checked). Linked stack: SV (PublisherController + SvPublisherInstance + SharedBuffer + EquationProcessor + FaultInjector + cid_generator + SvEncoder), the kit's MPMC ingest worker (vendored include/third_party/rigtorp/MPMCQueue.h — now USED by init()/submit*), GOOSE (GooseEncoder + GooseTxScheduler + GooseReceiver + GooseService + SpscBridge), asn1_ber_encoder, SvStats, PcapTx. Function-call flow for EQUATION SV (this is the EXISTING JS-FRONTEND path, NOT the kit's role — shown only for completeness of the shared backend): browser UI -> WebSocket -> PubWsServer -> PublisherController (configurePublisher + setPublisherEquations) -> SvPublisherInstance::prebuildFrames using EquationProcessor + sv_encoder -> SharedBuffer -> writer thread paces + npcap_send_packet. The kit re-exports the equivalent calls (publisher::add/configure/setEquations/start) as thin forwarders, but the simulator does not originate equations — so omit this flow from the kit/simulator DFD. Function-call flow for LIVE SV: net::open -> publisher::add -> configure -> setExternalMode(id,true) -> publisher::init() (spawns ingest worker + MPMC) -> start -> simulator threads call submitMagnitude(...) -> MPMC -> worker -> SvPublisherInstance::setLiveValue (scaled int32 latched into m_liveValue) -> writer thread reencodeFrame reads m_liveValue -> SharedBuffer frame bytes rewritten in place -> npcap_send_packet -> NIC -> shutdown/reset. Function-call flow for GOOSE: net::open -> goose::configureTx (registers the SPSC stream) -> goose::setValue (or publisher::submitBoolean -> worker -> goose::setValue) pushes a boolean into SpscBridge -> goose::startTx (spawns GooseTxScheduler which pulls the latched boolean via SpscBridge::sampleAt, encodes via goose_encode_frame, sends via npcap_send_packet with the IEC retransmit ramp); RX via goose::rxStart/rxRegister -> GooseReceiver decodes -> SpscBridge::pushOutbound -> goose::popDecoded. NOTE FOR DIAGRAMS / DFD: everything runs inside ONE process (the simulator app) — no external "teammate app" node. The live path has exactly one funnel stage: init() spawns one MPMC queue + one consumer worker; the simulator's many producer threads -> MPMC (multi-producer) -> single worker -> {SV magnitude => setLiveValue => SharedBuffer/reencodeFrame; GOOSE boolean => SpscBridge => GooseTxScheduler}. The SPSC bridge appears ONLY on the GOOSE branch; the SV live branch reaches the wire purely through the SharedBuffer/writer path. EQUATION SV is owned by the SEPARATE JS frontend (browser UI -> PubWsServer -> PublisherController) and must NOT be drawn inside the kit box — in a simulator-only DFD it is absent entirely; the kit box holds only the live-data nodes (init/worker+MPMC, submitMagnitude/submitBoolean, setExternalMode, net + GOOSE config). The only external endpoint is the NIC/wire.
